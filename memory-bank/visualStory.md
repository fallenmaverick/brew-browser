# Visual Story — brew-browser

**Owner:** Visual Storyteller
**Wave:** 3.5 — Visual narrative plan
**Date:** 2026-05-23
**Status:** Spec complete — ready for the project owner to shoot in ~30 minutes against a running `npm run tauri dev` window.

This file is the **shot list** for every visual asset the repo needs. The Technical Writer references the filenames + alt-text from §10 in the README; contributors drop new captures into the layout from §9. Nothing here gets generated by an agent — every asset is a real screenshot of the real app at 1100×720 on macOS Tahoe.

---

## 1. Narrative thesis (the one thing every visual must reinforce)

> **"It's a real macOS app, and it's really open."**

Every screenshot should make that legible in under two seconds. That means:

- The window chrome is stock macOS (traffic lights visible, no custom title bar gimmicks). Reads as native, not a webapp.
- The interior is dense, quiet, monochrome-with-amber-accent. Reads as utility, not marketing.
- There is real data on screen — actual installed packages, actual versions, actual streaming output. No `Lorem` rows, no placeholder names, no `example.com`.

The aesthetic anchor per `designSystem.md`: confident, plain, native macOS. Quiet and fast. Dense but readable. **Dark-first** — the hero is dark, the secondaries can be light for variety, but the primary visual identity is the dark theme with the warm amber brand at `oklch(72% 0.16 50)` on a `#1c1c1e` window background.

No drop-shadows on screenshots. No browser frames. No glowing edges. No tilted device mockups. No fake notification badges. The app is the asset.

### Composition checklist (what's visible)

Left sidebar:

- All five sections rendered (Library highlighted, Discover/Trending/Snapshots/Activity in default state).

- "Snapshots" shows snapshot count badge if any exist (seed 2–3 first per §6).

Library list (center):

- Filter chips visible: `All  Formulae  Casks  Outdated (N)`.
- 8–12 rows of real installed packages. **Variety matters** — include at least 2 casks among the formulae so the type pill differentiation is visible.
- At least one row should show an outdated chevron (`⟶ <new-version>`) so the outdated affordance reads.
- Recommended row mix (any of these you actually have): `bat`, `fzf`, `git`, `htop`, `iterm2` (cask), `jq`, `node`, `rectangle` (cask), `ripgrep`, `wget`. Family-feel of "developer Mac".

Detail panel (right):

- Package name at top, type pill (formula/cask).
- "Installed: X.Y.Z" + "Latest: X.Y.Z (up to date)" or "Upgrade available".
- One-line description (the real `brew info` description).
- Homepage link.
- Dependencies / Dependents rows (collapsed is fine — the disclosure triangle reads).
- Action button row at the bottom: `[Reinstall]` `[Uninstall]` — buttons clearly stylized per design system (Uninstall is red destructive).

### What NOT to include

- No modals open.
- No toasts visible.
- No mouse cursor on screen (use `Cmd+Shift+4` then space to capture window without cursor; or hit `defaults write com.apple.screencapture "include-pointer" -bool false` first).
- No browser DevTools open.
- No system clock / menu bar visible — screenshot the **window only**, not the full screen.

### Shoot instructions (≤2 minutes)

1. Launch the app dark-themed (`Cmd+Shift+L` until dark, or System Settings → Appearance: Dark).
2. Click `ripgrep` (or your chosen package) in Library. Panel slides in.
3. macOS: `Cmd+Shift+4`, then `Space`, then click the brew-browser window. Saves to Desktop at `brew-browser <timestamp>.png`.
4. Rename and move: `mv "~/Desktop/brew-browser*.png" /Users/michael/Clean/brew-browser/docs/screenshots/hero.png`.
5. Optional: run through `oxipng -o4 docs/screenshots/hero.png` to halve the file size losslessly. Skip if not installed.

## 4. Demo GIF / video — `docs/demo.gif` (and optional `docs/demo.mp4`)

### Goal

A **single ≤30-second** loop that shows the four-beat story: *open → search → install → done*. No narration, no captions, no cursor highlights — just the app being used. The intent: a visitor on the README sees the GIF auto-play, gets the entire product in one cycle, and either clicks "Build from source" or moves on. No middle ground.

### Scene-by-scene (target: 28 seconds total, comfortably under the 30s cap)

| # | Time | Frame | What happens |
|---|---|---|---|
| 1 | 0:00 – 0:03 | Library, dark, list of installed packages already on screen. | Hold for 3 seconds. Lets the viewer's eye land. **No motion.** This is the "what this is" beat. |
| 2 | 0:03 – 0:06 | Cmd+K → palette opens. | Palette overlay fades in. Empty input. Hold 1 second after open. |
| 3 | 0:06 – 0:10 | Type `wget` letter-by-letter into the palette. | Live filtering: installed match at the top, index matches below. ~250ms per keypress feels natural. |
| 4 | 0:10 – 0:12 | ↓ to highlight an index result, Enter. | Palette closes, detail panel slides in from the right with that package's info. |
| 5 | 0:12 – 0:14 | Hold detail panel. | Viewer reads the package name + description for a beat. |
| 6 | 0:14 – 0:16 | Click `[Install]`. | Button changes to `[Installing…]`. Activity drawer auto-opens at the bottom. |
| 7 | 0:16 – 0:24 | Drawer streams real brew output. | `==>` lines appear, download progress bar fills, "Pouring" line, final `🍺` line. Time-compress in editing if the real install is faster or slower than 8 seconds. |
| 8 | 0:24 – 0:27 | Toast appears bottom-right: "Installed wget (1.21.4)". Library reflects new package. Detail panel button row updates to `[Uninstall]` `[Reinstall]`. | Hold 3 seconds. |
| 9 | 0:27 – 0:28 | Last frame matches first frame (Library, no panel, no drawer) → loop seam. | Drawer collapses, panel closes, return to start state for clean loop. |

### Loop seam

The first frame (Scene 1) and the last frame (Scene 9) should be visually identical so the GIF loops cleanly. If they aren't, add a 200ms cross-dissolve in the editor to mask the cut.

### Tooling recommendation

**For the GIF (the README embed):**

- **[Kap](https://getkap.co/)** — free, open-source, macOS-native, exports MP4 and GIF cleanly. Use the "Crop to window" mode so the macOS window edges are pixel-perfect.
- Set frame rate to **15 fps** for the GIF (the human eye reads UI motion fine at 15; 30fps doubles file size for negligible gain on this kind of content).
- Capture at the actual 1100×720 logical size, **not** 2×. GIFs at Retina balloon the file size and add nothing.
- Export with **256-color palette** and **lossy** compression. Don't ship a true-color GIF — they're huge and unnecessary.

**For an optional MP4 (richer fallback, embedded via `<video autoplay loop muted playsinline>` if the Technical Writer wants):**

- **[Screen Studio](https://www.screen.studio/)** if you want polish (auto cursor smoothing, auto zooms). Paid app, optional.
- **OBS Studio** if you want free + open source. Slightly more setup.
- Export as **H.264 MP4, 1100×720, 30 fps, 2 Mbps bitrate**. Under 2 MB for a 28-second clip — well within GitHub's README limits.

### Size targets

| Format | Target | Hard cap |
|---|---|---|
| `docs/demo.gif` | ≤ 3 MB | 5 MB (GitHub renders GIFs up to ~10 MB, but anything over 5 MB feels rude on metered connections) |
| `docs/demo.mp4` (optional) | ≤ 2 MB | 5 MB |

If the GIF blows past 5 MB at 15fps/256-color/lossy, drop to **12 fps** and reduce the install scene (Scene 7) by trimming to a shorter window. Don't reduce dimensions below 1100px wide — the dense text becomes unreadable.

### Captions

**None.** The video has no audio and no on-screen text. The narrative is in the actions themselves. (If the Technical Writer disagrees and wants chapter markers, those go in the README prose around the embed, not inside the asset.)

### Alt-text (for the `<img alt="...">` attribute)

> `Short animated demo: opening brew-browser, searching for "wget" via Command-K palette, installing it with live streaming output, and seeing it appear in the Library — start to finish in under thirty seconds`

## 6. Branding / wordmark

### Recommendation: **No logo. Wordmark only.**

The brand is the word `brew-browser` set in the system font (SF Pro / `ui-sans-serif`) at semibold. That's it. Reasons:

- **Consistent with the design system narrative.** `designSystem.md` §1 explicitly rejects ornament. A bespoke logo for a six-tab utility undercuts the "honest, plain, no magic" tone.
- **Zero license cost, zero file to maintain.** A logo creates an asset that has to be redrawn at every size, audited for color drift between themes, replaced if the type changes. A wordmark in the system font has none of those problems.
- **Matches macOS-native peers.** Finder, Activity Monitor, Terminal — none of these have logos that read separately from their name. brew-browser sits in that family.
- **Future-proof.** If brand maturity ever demands a glyph (App Store listing, marketing site), it can be designed later. Shipping without one now costs nothing; shipping with a weak one we'd want to replace is expensive.

### What about the app icon?

Phase 5 still needs an actual `.icns` for the macOS dock — that's a Tauri build requirement, not a branding decision. The recommendation for the placeholder Phase 5 icon:

- A single character `b` set in SF Pro Heavy, color `oklch(72% 0.16 50)` (amber) on a dark `#1c1c1e` rounded-rect background, with macOS's standard icon mask radius (`squircle`, ~22.5% corner radius at 1024px).
- File: `src-tauri/icons/icon.png` and the generated `.icns`. Generate via `cargo tauri icon path/to/source-1024.png` from a single 1024×1024 PNG source.
- This icon does double duty as the dock icon AND the favicon for any future docs site.

That's out of scope for this visual story file (it's a build artifact, not a content asset) — flagged here only so the Technical Writer knows not to expect a separate logo file in `docs/`.

### Wordmark token

If anyone needs to render the wordmark in HTML or in another asset, the canonical recipe is:

```css
font-family: ui-sans-serif, -apple-system, "SF Pro", system-ui, sans-serif;
font-weight: 600;
letter-spacing: -0.01em;
color: var(--color-text-primary);
```

The amber brand color is **never** applied to the word `brew-browser` itself. The brand color is reserved for the active sidebar tab, primary buttons, the brew-mug icon (if added), and the social-card rule. Letting the wordmark stay neutral keeps it from feeling like a sticker.

```
    └── icons/                             # app dock icon (out of scope for this file)
        └── icon.png                       # Phase 5 build artifact
```

### Why `docs/` and not `assets/` or `images/`

- `docs/` is the GitHub convention for "stuff the README references" — keeps it visible in the repo root file tree, signals "this is part of the documentation surface."
- `assets/` reads like build-output / runtime-input (Vite, webpack convention) and would be confusing next to `src/` and `static/`.
- `images/` is too generic — `docs/screenshots/` is self-describing.

### A `.gitattributes` note

If GitHub starts complaining about the GIF size or git LFS becomes attractive, add:

```
docs/*.gif filter=lfs diff=lfs merge=lfs -text
docs/*.mp4 filter=lfs diff=lfs merge=lfs -text
```

For a single 3–5 MB GIF, LFS is overkill. Defer until the asset count grows or a single asset crosses ~10 MB. Flagged here so it doesn't surprise anyone later.

### Naming conventions

- All lowercase, hyphen-separated.
- Theme suffix (`-dark`, `-light`) when relevant.
- No version numbers in filenames — git history is the version control.
- No dates in filenames — git commit dates are the dates.
- Screenshots use `.png`. Animated demos use `.gif` (with optional `.mp4` companion). Static brand cards use `.png`. No JPEG anywhere — too much chroma compression on the dense UI text.
